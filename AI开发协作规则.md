# AI 开发协作规则

## 一、理解阶段：先看全貌，再动手

1. **改前端必读后端，改后端必读前端。** 绝不只看单层代码就动手。数据流是双向的——前端怎么调用、后端怎么返回、中间有什么转换，全部理清楚再改。

2. **涉及数据库表，先读 schema。** 确认每个字段的约束：NOT NULL、DEFAULT 值、外键 CASCADE 关系。避免生成 SQL 后因约束 violation 反复返工。

3. **存在同名 API 时，两个文件都要检查。** 例如 `/api/recommendations` 和 `/api/admin/recommendations` 可能共享逻辑但有不同查询，改一个不查另一个必然出 bug。

## 二、生成阶段：SQL 和数据脚本的强制检查清单

生成任何 INSERT/UPDATE/DELETE 语句后，逐项确认：

- [ ] 所有 NOT NULL 列都有值
- [ ] TRUNCATE CASCADE 会导致哪些表被级联清空（查 schema 外键）
- [ ] 子查询名称与 INSERT 名称完全一致，不存在拼写差异
- [ ] 同表内无重复数据（同名、同 ID、同组合）
- [ ] 数字字段在合理范围内（如 score 0-100）
- [ ] 特殊字符已转义（单引号 `''`，反斜杠 `\\`）
- [ ] 执行后立即抽查 2-3 条数据验证结果正确

## 三、交付阶段：验证完成前不清理

1. **生成脚本、抓取脚本在数据验证通过之前不删除。** 一旦数据有问题需要重新生成，重写脚本浪费时间。

2. **原始数据文件保留。** 从外部抓取的 JSON 数据是源头，可用于交叉验证和问题排查。

3. **只在最终交付时统一清理临时文件。** 中途清理极大概率导致返工。

## 四、通用禁止项

- **禁止生成 SQL 后立刻让用户执行。** 先过检查清单，确认无误再交付。

- **禁止使用 `Math.random()` 分配业务逻辑值。** 阶段分配、排序、分组等必须用确定性算法，结果可复现。

- **禁止在 API 中返回全量回退数据。** 如果没有实际匹配的推荐，就返回空，不要把所有行都塞进去凑数。前端自己处理空状态。

- **禁止跳过类型检查。** 每次代码修改后跑一次 `tsc --noEmit`，零错误才交付。

- **禁止假设"小改动不会有问题"。** 删一个 import、改一个字段名，都可能导致连锁错误。

## 五、数据生成规范

1. **命名不产生歧义。** 避免出现 `XX流流`、`变体` 等自造拼接词。数据来自外部源时，使用源数据的原始名称。

2. **去重在生成时完成。** 不依赖数据库 UNIQUE 约束兜底。同一实体同一关联只生成一条记录。

3. **默认值明确。** 不需要的字段设为 null，不用的字符串字段设为空字符串 `''`，不设 `"placeholder"` 或 `"TBD"` 等占位值。

4. **阶段/状态字段如果不使用，统一设为 null。** 不要用随机值或硬编码默认值填充。

## 六、前后端协同原则

1. **API 改返回值 → 追踪前端所有消费点。** 改了 `is_top`，就要查前端哪些地方用 `is_top` 做渲染判断。改了评分算法，就要查排序逻辑。

2. **API 返回多少数据，前端就展示多少。** 如果前端本来有分页，API 却只返回 3 条，那就是不匹配。

3. **筛选逻辑在后端做，排序逻辑可以前后端协作。** 不要让前端从 195 条数据里筛出 5 条——让 API 直接返回那 5 条。

## 七、开发节奏

1. **一个任务完成并验证后再开下一个。** 不要并行做"删功能"和"改算法"——出问题时无法定位根因。

2. **用户反馈 bug 后，先定位根因再动手。** 不要猜测。读代码、追踪数据流、找到具体哪一行导致的。

3. **修复 bug 后，检查是否引入新 bug。** 修 A 功能时，看 B 功能是否依赖了改动的代码。
